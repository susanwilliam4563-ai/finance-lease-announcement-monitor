# 融资租赁公告监控网站设计

## 第一性原理

这个网站的核心不是展示页面，而是降低漏报率和复核成本。系统必须回答四个问题：

1. 哪些主体发生了融资租赁相关事件？
2. 证据来自哪里，是否可复核？
3. 这件事是否值得优先关注？
4. 后续新增信息如何持续进入系统？

## 网站分层

### 数据源层

- 主源：巨潮资讯、交易所公告。
- 发债主体源：债券公告披露源，后续接入。
- 复核源：公司官网。
- 补充源：预警通。
- 兜底源：互联网搜索、公众号等公开线索。

当前实现巨潮资讯主源、外部线索检索入口、统一 JSON 导入契约；其他来源不伪装为已自动抓取，必须通过 `source_class` 和 `review_status` 显示可信度。

### 事件识别层

先用强关键词召回：

- 融资租赁
- 融资性售后回租
- 售后回租
- 售后租回
- 金融租赁
- 融资租赁合同
- 融资租赁业务
- 租赁融资

再用正文证据确认：

- 承租人
- 出租人
- 租赁物
- 租赁期限
- 融资金额
- 担保
- 抵押/质押
- 关联交易

### 主体画像层

A 股上市公司按股票代码匹配公开公司资料：

- 地区：上市公司注册地省份。
- 行业：证监会行业分类。
- 来源：东方财富公司资料公开接口。

这些字段用于覆盖早期的规则推断结果。未匹配的少数特殊代码、北交所代码或发债主体继续标记为 `待补充`，等待正式主体库或预警通/Wind/Choice 导出数据补齐。

### 数据沉淀层

使用本地 SQLite 保存标准化记录。每条记录包含主体、公告、金额、期限、交易对手、证据片段、风险标签和复核状态。

历史起点统一为 `2021-01-01`。数据库为空时从这个日期回填；数据库已有数据时默认抓取最近 14 天做增量更新。

历史回填采用两阶段策略：

1. 快速索引：先抓取标题、链接、元数据和命中关键词，不下载全部 PDF，保证网站尽快具备 2021 起检索能力。
2. 正文补全：对新增、A/B 级、待复核或用户关注的记录再下载 PDF 抽取正文，补证据链。

### 看板层

看板必须支持：

- 全文关键词搜索。
- 按年度、季度、自定义日期、主体类型、地区、行业、来源层级、具体来源、风险标签、复核状态、关注等级筛选。
- 查看公告证据链和来源链接。
- 生成外部公开搜索入口，用于公众号、公司官网和互联网线索拓展。
- 导出当前筛选结果。
- 手动刷新、历史回填和自动增量刷新。

## 关注等级

- A：售后回租、关联交易、担保/抵押、大额融资等多项并存。
- B：明确融资租赁事件，金额较大或信息披露不完整。
- C：普通融资租赁交易或进展公告。
- D：弱线索或仅留档记录。

## 重大风险

1. 数据源接口变化，导致抓取失败。
2. 关键词策略老化，导致漏报。
3. 主体关系库缺失，导致地区、行业、集团归属不准。
4. 发债主体来源尚未接入，导致当前第一版偏 A 股公告。
5. 预警通存在登录和导出限制，不宜直接作为无授权自动爬虫源。
